# Build gotchas: PythonTeX + latexmk + biber + the didactic submodule

Hard-won fixes for getting a `makefiles`-based document (the shared `tex.mk` /
`latexmkrc` build, with PythonTeX, biblatex/biber, minted, and the `didactic`
submodule) to compile on current TeX Live. Search this file when a `make`
build of such a project fails.

## minted v3 (TeX Live 2024+)

The `[outputdir=...]` option to `minted` was **removed** and is now a fatal
error; load plain `\usepackage{minted}` (output dir auto-detected). Also,
minted v3 rejects a `minted`/`\mintinline` language argument that is split
across a line break (`Pygments lexer " python" is unknown`). See the
**latex-writing** skill's `references/minted-v3-and-floats.md`.

## PythonTeX needs the makefiles `latexmkrc` (the "mylatex" fudge)

When LaTeX runs with an output directory (`-output-directory=ltxobj`), the
PythonTeX custom dependency is not detected unless latexmk uses the
`mylatex`/symlink fudge shipped in the makefiles `latexmkrc`. Symptom without
it: latexmk falls back to `-use-make` and tries to `make` the `.bbl`/`.toc`
itself (see biber section), and PythonTeX never runs cleanly.

Create the symlink in the project (it is **not** auto-created):

```bash
make latexmkrc          # symlinks ./latexmkrc -> ./makefiles/latexmkrc
```

### `latexmkrc` is generated and should be gitignored

`latexmkrc` in the makefiles repo is **tangled from `tex.mk.nw`** (chunk
`<<[[latexmkrc]]>>`, rule `latexmkrc: tex.mk.nw`, `OTHERS += latexmkrc`). Like
the other generated helper `gitattributes`, it should be **gitignored and
untracked** in the makefiles repo — fix the source in `tex.mk.nw`, not the
generated file. In the *consuming* project, the `./latexmkrc` symlink should be
gitignored too.

### Invoke `pythontex` portably (not `pythontex3`)

The shipped `latexmkrc` cus-dep historically called `pythontex3`, which may not
exist (TeX Live's wrapper is `pythontex`, and its shebang is `#!/usr/bin/env
python` — `python` may be absent). Match `tex.mk`'s portable form in the
`pythontex` sub of `tex.mk.nw`:

```perl
return system("python3 \$(which pythontex) --interpreter python:python3 --verbose \"$_[0]\"");
```

(`python3 $(which pythontex)` runs it under python3 regardless of the shebang;
`--interpreter python:python3` makes the executed PythonTeX code use python3.)

## PythonTeX output/working directories — don't write outside the repo

`\setpythontexoutputdir` and `\setpythontexworkingdir` are **relative to the
main document's directory**, not to wherever the build happens to run. With the
`latexmkrc` mylatex fudge, LaTeX runs from the **project root**, so
`\setpythontexworkingdir{..}` resolves to the **parent of the repo** and scatters
`didactic_output_*.txt` / PythonTeX temp files *outside* the project. Match the
setting to your layout:

- **Flat layout** (root `.tex`, build from root): use
  `\setpythontexworkingdir{.}` (project root). `{..}` writes outside the repo.
- **`doc/` subdir layout** (build from `doc/`): `{..}` correctly points at the
  project root.

Gitignore the PythonTeX artifacts the build leaves in the working/output dir:
`*.pytxcode *.pytxmcr *.pytxpyg pythontex_data.pkl pythontex-files-* py_*.stdout
didactic_output_*.txt` (and the `latexmkrc` symlink).

## biber vs bibtex: `latexmk -use-make` can pick the wrong `.bbl` rule

`tex.mk` defines **two** `%.bbl` rules — one from `%.aux` (bibtex/`bibtexu`) and
one from `%.bcf` (biber). With `latexmk -pdf -use-make`, when LaTeX reports
"No file doc.bbl", latexmk delegates `make doc.bbl`, and GNU make picks the
**first** matching rule (bibtex) — fatal for a biblatex+biber project:

```
I found no \bibdata command---while reading file ltxobj/doc.aux
make: *** [.../tex.mk: ltxobj/doc.bbl] Error 2
```

The robust fix is to use the `latexmkrc` path (above), which lets latexmk run
biber natively instead of delegating to make. If you ever drop the `latexmkrc`,
also drop `-use-make` (`PDFLATEX = latexmk -pdf -8bit` in the project Makefile)
so latexmk handles biber and `.toc` reruns itself.

## The `didactic` submodule: `didactic.sty` is generated

The build loads `didactic.sty` via a root symlink to the **submodule**
(`didactic/didactic.sty`), not the copy in `~/texmf`. `didactic.sty` is
generated by docstrip from `didactic.dtx` and is **untracked** in the submodule.
After updating the submodule, regenerate it — and note docstrip refuses to
overwrite interactively, so remove it first:

```bash
cd didactic && rm -f didactic.sty && make didactic.sty   # NOT `make -B`, which prompts
grep -m1 'didactic\]' didactic.sty                        # confirm the new version
```

The Makefile rule depends only on `didactic.ins` (unchanged across versions), so
plain `make` reports "up to date" even when the `.dtx` changed — delete and
rebuild.

## Clean state matters

These multi-pass builds (latex → biber → pythontex → latex×N) leave a lot of
intermediate state. When chasing an intermittent error (e.g. minted "Cannot find
input file `didactic_output_*.txt`", which is a stale-state symptom, *not* the
same as "Float(s) lost"), do a **full** `make clean` plus removal of the root
PythonTeX artifacts (and any parent-dir pollution) before re-testing — partial
`rm ltxobj/doc.*` leaves inconsistent symlinks/`.pytxmcr` that produce
misleading errors.
